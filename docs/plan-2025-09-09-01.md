# Plan: Stage-based converter refactor

Date: 2025-09-09
Author: automated coding agent

Overview
========
This document outlines a stage-based (multi-pass) refactor of the converter in
`splurge-unittest-to-pytest`. The goal is to make transforms deterministic,
easier to test, and robust against ordering and reference bugs (imports, fixture
cleanup, self->fixture attachment). We treat each pass as a "stage" with a
clear contract.

High-level goals
----------------
- Deterministic insertion of `import pytest` before any fixture decorators.
- Safe fixture generation from `setUp` assignments (yield vs return, local binding).
- Robust detection of cleanup statements (nested conditionals and blocks).
- A clear, testable pipeline: Collector → Generator → Injector → Rewriter → Validator.
- Minimal risk incremental migration with unit tests for each stage.

Stage list (summary)
--------------------
0. Parse & MetadataWrapper (optional)
1. Collector (read-only Visitor)
2. Normalizer (small, safe transforms)
3. FixtureGenerator (create FunctionDefs)
4. ImportInjector (add pytest import deterministically)
5. FixtureInjector (insert fixtures + autouse attach fixture)
6. TestMethodRewriter (remove self/cls, add fixture params)
7. PostValidator (sanity checks)
8. Tidy & Formatting (blank lines / style)

Data shapes and contracts
-------------------------
- CollectorOutput (dataclass):
  - has_unittest_content: bool
  - module_docstring_index: int | None
  - imports: list[ImportInfo]
  - classes: dict[class_name -> ClassInfo]

- ClassInfo (dataclass):
  - node: cst.ClassDef
  - setup_methods, teardown_methods, test_methods: list[cst.FunctionDef]
  - setup_assignments: dict[attr_name -> cst.BaseExpression]
  - teardown_statements: list[cst.BaseStatement]

- FixtureSpec (dataclass):
  - name, value_expr, cleanup_statements, yield_style, local_value_name, fixture_node

Stage-by-stage details
----------------------

Stage 0 — Parse & MetadataWrapper
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
- Optional: wrap module in libcst.MetadataWrapper to enable scope/qualified-name
  providers if needed later.
- Output: the wrapper or original module placed in StageContext.

Stage 1 — Collector
~~~~~~~~~~~~~~~~~~~
Purpose
- Read-only traversal to collect facts: setUp assignments, tearDown cleanup,
  test methods, imports, docstring position, and a flag whether unittest is used.

Behavior
- For every ClassDef, find methods matching setup/teardown/test patterns.
- Record self.<attr> assignments found in setUp bodies.
- Record teardown statements (entire statements, not transformed yet) and
  optionally run them through a small SelfReferenceRemover to simplify later
  pattern matching.

Outputs
- CollectorOutput dataclass containing per-class ClassInfo objects.

Tests
- Verify that setUp assignments and nested cleanup If blocks are captured.

Notes
- Collector must not mutate nodes. Keep it robust and defensive: log and
  skip unexpected shapes instead of throwing broad exceptions.

Stage 2 — Normalizer
~~~~~~~~~~~~~~~~~~~~
Purpose
- Apply small, safe normalizations: canonicalize method names, optionally
  remove self/cls prefixes in collected bodies to make matching simpler.

Behavior
- Transform limited regions (collected statements) to a normalized form.

Tests
- Ensure normalization doesn't change unrelated code.

Stage 3 — FixtureGenerator
~~~~~~~~~~~~~~~~~~~~~~~~~~
Purpose
- From CollectorOutput, create FixtureSpec entries and generate `cst.FunctionDef`
  fixture nodes. Do not insert them into module yet.

Behavior
- For each setup assignment:
  - If cleanup exists -> yield-style fixture
  - If value_expr is a simple literal (Integer/Float/SimpleString) yield literal
    directly; otherwise bind to local name `_<attr>_value` and rewrite cleanup to
    reference the local name via a small ReplaceName transformer.
- Ensure unique fixture names; append suffix on collisions.

Tests
- Validate fixtures for: literal yield preservation, local binding + cleanup
  rewrite, uniqueness handling.

Stage 4 — ImportInjector
~~~~~~~~~~~~~~~~~~~~~~~~
Purpose
- Ensure `import pytest` is present and inserted after module docstring and
  before fixtures/decorators.

Behavior
- Detect existing pytest import and avoid duplicates.
- Insert pytest import at pos: after docstring if present, else after other
  imports, else before first non-import statement.

Tests
- Documented insertion cases (docstring+imports, no docstring, pytest already
  present) covered.

Stage 5 — FixtureInjector
~~~~~~~~~~~~~~~~~~~~~~~~~
Purpose
- Insert generated fixture nodes (and optional compat autouse fixture) into the
  module body at the deterministic position chosen by ImportInjector.

Behavior
- Insert an EmptyLine then the fixture nodes one-by-one.
- Add `_attach_to_instance` autouse fixture when compat enabled and collector
  indicates unittest usage.

Tests
- Verify fixtures exist and are in expected order; verify autouse fixture
  attaches the right fixture names.

Stage 6 — TestMethodRewriter
~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Purpose
- Remove `self`/`cls` parameters from test methods and add fixture params
  based on created fixtures.

Behavior
- For test methods: create new param list with fixture params appended (or
  inferred/filtered) and remove first param if `self`/`cls` as appropriate.
- Use SelfReferenceRemover on test body if we decide to strip interior `self.`
  references (still rely on compat autouse when enabled to attach values).

Tests
- Ensure test methods reference fixtures correctly and remain syntactically
  valid when run under pytest.

Stage 7 — PostValidator
~~~~~~~~~~~~~~~~~~~~~~~
Purpose
- Lightweight sanity checks: pytest import exists if fixtures were generated,
  no duplicate fixture names, parsed module compiles.

Behavior
- Parse the generated module string with libcst.parse_module (or just
  re-serialize and parse) to ensure validity, and report quick errors.

Stage 8 — Tidy & Formatting
~~~~~~~~~~~~~~~~~~~~~~~~~~~
Purpose
- Minor spacing/EmptyLine placement for readability.

Behavior
- Insert/remove EmptyLine nodes to create separation between fixtures and
  classes/functions.

StageManager & API
------------------
- Implement `StageManager` that holds an ordered list of stage callables. Each
  stage accepts a `StageContext` (module, wrapper, metadata) and returns a
  modified `StageContext`.
- Provide two modes: `fail-fast` (CI/CLI) and `best-effort` (library/CLI with
  --safe). For now fail-fast is the default.

Testing strategy
----------------
- Unit tests per stage (Collector, FixtureGenerator, ImportInjector, Rewriter).
- Golden snapshot tests for a small set of example files (to reduce regression
  risk during refactor).
- Integration test (convert + run pytest on converted output) as an optional
  separate job in CI.

Edge cases & mitigation
-----------------------
- Dynamic assignments (setattr): collector records as dynamic; generator may
  skip and log warning.
- Nested cleanup (If/try/finally): collector records whole statements; generator
  attempts safe replacement, else preserves original and emits a warning.
- Name collisions: generator appends suffix; validator ensures collisions do not
  cause accidental overwrite.

Incremental migration plan (low risk)
------------------------------------
1. Implement Collector with unit tests and wire StageManager to call it.
2. Implement FixtureGenerator and unit tests; integrate but don't inject.
3. Add ImportInjector + FixtureInjector and unit tests; start inserting fixtures
   but keep old transformer as fallback.
4. Implement TestMethodRewriter and run full test suite; fix regressions.
5. Add PostValidator and Tidy stage; finalize refactor and remove old code.

Deliverables

- Unit tests under `tests/unit/` for each stage and `tests/integration/` for
