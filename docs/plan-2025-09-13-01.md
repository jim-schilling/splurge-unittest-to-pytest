Documenting the generator overhaul plan and acceptance criteria.

Overview
--------
This plan describes a clean, testable rewrite of the generator stage used by
the unittest->pytest converter. The existing `stages/generator.py` contains
many heuristics and ad-hoc rewrites; `generator_v2` will provide a smaller,
well-documented core and clear helpers that produce the same public output
shape used by downstream stages (specs and fixture_nodes).

Historical note
---------------
The compatibility-mode and engine selection options (for example, `--no-compat` and `engine=`) were removed in release 2025.1.0. This plan documents work that predates that change and is retained for historical reference.

Inputs/Outputs
--------------
- Input: pipeline `context` dict containing `collector_output` (CollectorOutput)
  and optional `module` (libcst.Module) and flags (e.g., `autocreate`).
- Output: dict with keys `fixture_specs` (mapping), `fixture_nodes` (list of
  libcst nodes), optional `needs_typing_names` (sorted list), and
  `needs_shutil_import` (bool) when shutil is used.

Acceptance criteria
-------------------
1. Deterministic behavior: given the same `CollectorOutput` the produced
  fixtures and typing needs must be stable.
2. No composite groupings: this project will avoid synthesizing composite
  fixtures (like `temp_dirs`). Instead the generator will emit one fixture
  per recorded attribute. This keeps the converter simple and makes outputs
  easier to reason about and test.
3. mkdtemp and other tempfile usages are preserved exactly; no automatic
  conversion to `tmp_path` or stripping of calls.
4. Non-dir attributes (e.g., `main_config`, `config_file`, `mock_*`) are
  emitted as per-attribute fixtures with appropriate yield/return shapes.
5. No hard-coded source-snippet insertions; all nodes built programmatically
  using libcst builders.


Plan
----
- Implement small helpers: build_local_assignments, rewrite_cleanup,
  emit_per_attr_fixture_node. The implementation will intentionally avoid any
  composite/grouping helpers.
- Add unit tests for each helper and integration-style tests that invoke
  `generator_v2` on representative CollectorOutput instances and check
  high-level properties (one fixture per attribute, mkdtemp preserved,
  cleanup attached to the correct fixture).
- Iterate against `tests/unit/test_converter_goldens_strict.py` until green.

Notes
-----
This plan intentionally keeps the v2 generator isolated (module suffix _v2)
so downstream stages can be migrated incrementally.
# Converter Tweak Plan — Unittest → Pytest (2025-09-13)

## Summary

This plan specifies a prioritized, actionable set of changes to the converter pipeline in this repository to: improve fixture generation (setUp/tearDown → fixtures), canonicalize fixture decorator/return forms, expand typing inference and import injection, preserve patch/patch-order semantics, and add tests/goldens to lock the behavior.

Scope: changes limited to the generator and converter stages, import injector, and test goldens. Implementation should preserve backwards compatibility where reasonable and prefer small, testable steps.

## Acceptance criteria

- The converter emits `@pytest.fixture` (no parentheses) when there are no decorator args.
- Fixtures that require cleanup are emitted as generator fixtures using `yield` inside a try/finally block with deterministic typing: `Generator[<ReturnShape>, None, None]` and a single `from typing import ...` import for typing names used.
- Simple literal/container/comprehension fixture values become direct `return <value>` with an accurate annotation when possible.
- setUp/tearDown pairs that operate on the same `self.<attr>` set produce a single fixture that creates resources and performs the teardown in finally.
- Patch decorators are preserved with the same order and the produced function accepts patch mocks as parameters in the same left-to-right mapping.
- Tests: add golden/unit tests (one per sample) asserting the produced pytest source matches the sample-pytest canonical file.
  - NOTE: Unit tests for `import_injector` (typing/pathlib insertion, merge, ordering, tidy compatibility) were added under `tests/unit/test_import_injector.py` as part of this work.

## High-level design decisions

- Central typing collection: keep the generator-stage `_infer_ann` (already present) to return (annotation_node, set_of_typing_names). The import_injector will consume the union-of-all-typing-names and add a single `from typing import ...` ImportFrom sorted deterministically.

- Canonical decorator form: produce an Attribute decorator node for plain `@pytest.fixture` (no args), and a Call when args exist. This ensures deterministic printed form.

- Yield vs return policy:
  - If cleanup is required or the fixture value is not a simple literal/container/comprehension, emit a generator fixture using yield inside try/finally.
  - If the RHS is a simple value (literal, container of simple values, or comprehension) and no teardown is needed, emit direct `return <value>` with an inferred annotation.

- Naming: generate unique internal local names for bound values using a deterministic pattern `_tmp_<base>` with numeric suffixes to avoid collisions.

## File-level tasks and concrete edits

1) Generator: enhance setUp/tearDown recognition and fixture emission
- File: `splurge_unittest_to_pytest/stages/generator.py`
- Tasks:
  - Extend the setUp parser to collect all `self.<attr> = <expr>` assignments and their RHS expressions.
  - Parse tearDown to collect cleanup operations that reference `self.<attr>` (e.g., shutil.rmtree(self.temp_dir), .close()) and map them to attributes.
  - If setUp assigns multiple related attributes and tearDown cleans them, emit a single fixture function that creates local variables for each attribute, yields an aggregate (dict) of them, and performs cleanup in finally. Example shape follows `sample-04-pytest.txt`.
  - Reuse `_infer_ann` to annotate the fixture return type as `Generator[Dict[str, Path], None, None]` or `Generator[Path, None, None]` where appropriate and collect typing names.
  - Ensure no None entries are appended to IndentedBlock; append only valid libcst BaseStatement nodes.
- Tests: add unit tests to assert expected function names, docstrings, and body shapes for several sample classes.

2) Fixture emission helpers and decorator canonicalization
- Files: `splurge_unittest_to_pytest/converter/fixtures.py`, `splurge_unittest_to_pytest/converter/decorators.py`
- Tasks:
  - Update decorator builder: when no args/kwargs, return cst.Decorator(cst.Attribute(cst.Name('pytest'), cst.Name('fixture'))).
  - Update fixture emission to prefer direct `return <literal>` for simple RHS cases (using `_is_simple_value` predicate). Otherwise produce yield fixtures when cleanup or complexity requires binding.
  - Ensure annotations from `_infer_ann` are attached to FunctionDef. When a fixture returns a dict-of-paths, use `Dict[str, Path]` (collect `Dict` into typing imports and leave `Path` as a normal reference to `pathlib.Path`).

3) Typing import injection
- File: `splurge_unittest_to_pytest/stages/import_injector.py`
- Tasks:
  - Add a deterministic ImportFrom: `from typing import Name1, Name2, ...` with names sorted.
  - Avoid duplicating imports if a typing import is already present. Position the typing import after stdlib imports and before local imports, following current insertion heuristics.

4) Patch/decorator lifting and signature ordering
- Files: `splurge_unittest_to_pytest/stages/generator.py` (or converter path where methods are converted)
- Tasks:
  - When converting test methods with decorators like `@patch`, lift decorators to the function level and ensure the produced function parameters include the patched mocks first in the order specified by decorators.
  - Preserve decorator strings unchanged (exact import path strings inside patch) to avoid changing semantics.

5) Naming collision resolution
- File: `splurge_unittest_to_pytest/stages/generator.py`
- Tasks:
  - Implement `_choose_unique_name(base, used_names)` that returns a deterministic unique name using suffixes if necessary.
  - Use this name when emitting temporary binds for complex fixtures.

6) Tests & goldens
- Files to add/update: `tests/data/*_converted.expected` or `tests/data/samples/*-pytest.txt` comparisons; unit tests in `tests/unit/`.
- Tasks:
  - For each sample in `tests/data/samples`, add a unit test that runs the converter pipeline and asserts that the produced code exactly matches the `*-pytest.txt` canonical file. Use a helper to normalize trailing whitespace and import ordering if needed.
  - Update any relaxed tests to assert the canonical form strictly (or produce separate golden files for canonical output). This locks the new canonical behavior.

7) Lint & type checks
- Files: whole repo
- Tasks:
  - After code edits run `ruff check --fix .` and fix any remaining F841 (unused variables) issues.
  - Run mypy (project config) and iterate to fix real issues (limit scope to source files first; do not attempt to silence test typing noise unless necessary). Add small Optional annotations where generator stage legitimately assigns None temporaries.

## Examples (small patterns)

- Canonical decorator (no args): `@pytest.fixture`
- Generator fixture with cleanup:

```python
@pytest.fixture
def temp_dirs() -> Generator[Dict[str, Path], None, None]:
    temp_dir = Path(tempfile.mkdtemp())
    try:
        yield {"temp_dir": temp_dir}
    finally:
        shutil.rmtree(temp_dir, ignore_errors=True)
```

- Simple direct-return fixture:

```python
@pytest.fixture
def test_file(temp_dir: Path) -> Path:
    return temp_dir / "test.txt"
```

## Edge cases and mitigations

- Complex setUp logic (control flow, helper functions) -> conservative approach: emit a yield fixture and include a small wrapper to preserve semantics.
- Cleanup that references multiple attributes or external state -> keep multi-attribute fixture that yields a dict to avoid reordering problems.
- Tests that assert exact textual layout: update goldens to canonical output; for safety, add a relaxed comparator helper used in CI to surface only semantically-significant differences.

## Risks

- Over-eager inference could emit incorrect annotations; mitigate by falling back to `Any` or no annotation when ambiguous.
- Tests referencing exact whitespace/order may fail; mitigate by updating goldens and using deterministic import ordering.
